## 2. 当前整体逻辑

### 2.1 本地角色选择流程

当前本地角色选择由 `gameStore.selectPhase` 控制，只存在两个状态：

- `p1`：玩家1选择角色。
- `p2`：玩家2选择角色。

当前 `CharacterSelect.tsx` 的选择逻辑如下：

1. `selectPhase === 'p1'` 时，点击角色卡会调用 `setPlayer1Character(char)`。
2. 玩家1选中后，页面出现「玩家2选择」按钮。
3. 点击「玩家2选择」后，通过 `setSelectPhase('p2')` 切换到玩家2编辑态。
4. `selectPhase === 'p2'` 时，点击角色卡会调用 `setPlayer2Character(char)`。
5. 玩家1、玩家2都存在后，页面出现「开始战斗」按钮，点击后调用 `initBattle()`。
6. 选完两个角色后，可以点击「重新选择玩家1」或玩家1/玩家2预览卡切换编辑对象。
7. 点击「清空选择」会清空本地选角并回到玩家1编辑态。

### 2.2 本地战斗初始化

`gameStore.initBattle()` 会读取 `player1Character` 和 `player2Character`，创建两名战斗角色：

- 玩家1初始位置在左侧，`facingRight: true`。
- 玩家2初始位置在右侧，`facingRight: false`。
- 初始化完成后进入 `screen: 'battle'`。

当前战斗页「返回选择」会先将 `selectPhase` 设回 `p1`，再执行 `setScreen('select')`，避免返回后仍停留在玩家2编辑态。

### 2.3 本地移动与朝向

当前本地移动逻辑在 `src\game\engine.ts`：

- 玩家1：`A` 左移，`D` 右移，`W` 跳跃。
- 玩家2：方向键左/右移动，方向键上跳跃。
- 角色移动速度读取 `fighter.character.moveSpeed`。
- 当前代码已改为：按左就面朝左，按右就面朝右。
- 当前代码已取消：背向对手移动自动变为格挡并降速。

远程技能方向依赖 `attacker.facingRight`：

- `facingRight === true`：弹道向右。
- `facingRight === false`：弹道向左。

因此朝向是否正确，会直接影响远程攻击方向。

### 2.4 在线对战移动与朝向

Socket.IO 服务端权威逻辑在 `D:\allproject\backendMinigame\src\game\engine.ts`。服务端会按固定 tick 接收双方输入，更新位置、朝向、技能和弹道，再广播快照给前端。

当前服务端移动逻辑也已改为：

- 按左时 `fighter.facingRight = false`。
- 按右时 `fighter.facingRight = true`。
- 不再因为「远离对手」自动进入格挡降速。

在线模式如果仍看到旧行为，需要确认后端服务是否已经重启，避免运行的仍是旧编译结果或旧进程。

## 3. Bug 1：选完两个角色后不能重新选择玩家1

### 3.1 现象

本地角色选择页中，玩家1选择角色后切到玩家2选择。玩家2也选择完成后，只能开始战斗，无法方便地回到玩家1重新选择。

玩家感受是：选错玩家1后没有回退入口，流程被锁住。

### 3.2 修复前根因

`selectPhase` 是单向流程：

- 页面提供了 `p1 -> p2` 的按钮。
- 没有提供 `p2 -> p1` 的按钮。
- `PlayerPreview` 只展示当前玩家，不支持点击切换编辑对象。
- `gameStore` 没有封装 `setSelectPhase` action，组件里直接使用 `useGameStore.setState({ selectPhase: 'p2' })`，导致流程能力不完整。

同时，`返回选择` 只切 screen，不重置选择阶段：

- 战斗页「返回选择」调用 `setScreen('select')`。
- `player1Character`、`player2Character`、`selectPhase` 会保持旧值。
- 如果上一次停留在 `p2`，再次进入选择页仍可能处于玩家2阶段。

### 3.3 影响

- 选错玩家1后必须返回标题或刷新页面，体验割裂。
- 本地双人轮流选角时容易误以为页面卡住。
- 后续如果加入更多选项，比如皮肤、地图、回合制设置，单向状态会继续放大问题。

### 3.4 已落地修复方案

已按方案 A 落地。在 `gameStore` 增加显式 action：

```ts
setSelectPhase: (phase: 'p1' | 'p2') => void;
clearSelections: () => void;
```

在 `CharacterSelect.tsx` 增加：

- 当前阶段是玩家2时，显示「重新选择玩家1」。
- 玩家1预览卡可点击，点击后进入 `p1`。
- 玩家2预览卡可点击，点击后进入 `p2`。
- 角色卡显示「移速」。
- 详情面板显示「移动速度」数值和速度条。
- 动作区显示「清空选择」。

当前同角色冲突策略：

- 玩家1编辑态下，玩家2已选角色会禁用。
- 玩家2编辑态下，玩家1已选角色会禁用。
- store 内也保留了同角色兜底保护，避免未来入口绕过 UI 时出现重复选择。

#### 后续可选方案：更稳定的状态模型

将角色选择改成「显式编辑目标」模型：

```ts
type SelectSlot = 1 | 2;

interface GameStore {
  selectingSlot: SelectSlot;
  setSelectingSlot: (slot: SelectSlot) => void;
  setCharacterForSlot: (slot: SelectSlot, character: Character) => void;
}
```

这样 UI 不再依赖 `p1/p2` 这种流程阶段，而是明确知道当前正在编辑哪个玩家。

优点：

- 以后加入 3P、4P、AI、联机房间槽位时更容易扩展。
- 「点击谁的头像就编辑谁」更符合用户直觉。
- 可以减少组件里直接 `setState` 的情况。

### 3.5 验收标准

- 玩家1选完后可以切到玩家2。
- 玩家2选完后可以切回玩家1。
- 玩家1重新选择后，预览区和详情区立即更新。
- 玩家1、玩家2不能误选同一个角色，除非产品明确允许镜像对战。
- 从战斗页返回选择页时，状态符合产品预期：
  - 如果保留上次选择，应显示清楚当前正在编辑谁。
  - 如果重开选择，应清空两名角色并回到玩家1。

## 4. Bug 2：左右移动有时变慢

### 4.1 现象

玩家左右移动时，有时速度明显变慢；不同位置、不同方向表现不一致。

### 4.2 历史根因

旧逻辑把「远离对手移动」当作格挡：

```ts
const awayHeld = fighter.facingRight ? leftHeld : rightHeld;
const towardHeld = fighter.facingRight ? rightHeld : leftHeld;

if (awayHeld && !towardHeld) {
  dx *= 0.4;
}

fighter.isBlocking = awayHeld && !towardHeld;
```

同时旧逻辑还会根据双方位置自动修正朝向：

```ts
if (state.p1.x < state.p2.x) {
  state.p1.facingRight = true;
  state.p2.facingRight = false;
} else {
  state.p1.facingRight = false;
  state.p2.facingRight = true;
}
```

这两个逻辑叠加后，用户按左/右时实际发生的是：

- 如果这个方向是远离对手，速度被乘以 `0.4`。
- 如果这个方向是靠近对手，保持正常速度。
- 当双方交换站位时，朝向自动翻转，慢速方向也随之变化。

所以玩家感受到的是「左右移动偶尔变慢」，本质是移动键承担了两个职责：移动和格挡。

### 4.3 当前代码状态

本地和服务端移动逻辑已调整为输入决定朝向：

- 按左：向左移动，朝左。
- 按右：向右移动，朝右。
- 移动不再自动触发格挡。
- 远程技能根据当前输入朝向发射。

前端本地：`D:\allproject\minigame\src\game\engine.ts`

服务端在线：`D:\allproject\backendMinigame\src\game\engine.ts`

### 4.4 剩余风险

前端本地 `src\game\engine.ts` 中未调用的 `updateFacing(state)` 遗留函数已删除，后续不容易误把自动朝向逻辑重新加回 tick。

### 4.5 验收标准

- 玩家1按 `A` 持续左移，速度稳定。
- 玩家1按 `D` 持续右移，速度稳定。
- 玩家2按 `ArrowLeft` 持续左移，速度稳定。
- 玩家2按 `ArrowRight` 持续右移，速度稳定。
- 双方交换站位后，左右移动速度仍稳定。
- 移动不会自动显示格挡状态。

## 5. Bug 3：一个角色快，一个角色慢

### 5.1 现象

玩家感觉不同角色左右移动速度差异很大，容易误认为是移动 bug。

### 5.2 根因

这不是随机 bug，而是角色配置里的 `moveSpeed` 差异：

| 角色 | moveSpeed |
| --- | ---: |
| 小明 | 4 |
| Emily | 5 |
| 张总 | 2 |
| 小红 | 4 |
| 阿杰 | 6 |
| 美美 | 4 |

其中张总移速最低，阿杰移速最高。两者速度差为 3 倍。

问题在于角色选择页目前主要展示生命、蓝量、攻击速度等信息，没有把「移速」作为显性属性展示出来。用户不知道角色速度差是设计还是异常。

### 5.3 产品判断

这里有两个方向，需要明确产品取舍。

#### 方向 A：保留角色差异

如果目标是轻格斗/派对对战，可以保留移速差异，但必须在角色选择页展示「移速」。

建议展示：

- 生命
- 蓝量
- 攻速
- 移速
- 定位标签，例如「高机动」「重装」「远程」「爆发」

同时需要重新平衡数值，避免张总 `2` 和阿杰 `6` 的差异过大导致对战体验失衡。

#### 方向 B：统一基础移速

如果目标是公平竞技手感，建议所有角色统一基础移速，比如全部为 `4` 或 `4.5`，再用技能机制体现差异。

例如：

- 阿杰通过技能获得短时加速。
- 张总通过护盾/控制补强。
- Emily 通过远程能力建立优势。

### 5.4 推荐方案

短期建议走方向 A：

1. 保留现有角色差异。
2. 在角色选择页补充「移速」显示。
3. 将 `moveSpeed` 差异收窄，例如范围控制在 `3.5 - 5.0`。
4. 在调试阶段加入速度可视化，方便确认是配置差异还是逻辑 bug。

后续如果要做更公平的排位/联机匹配，再切换到方向 B。

### 5.5 验收标准

- 角色选择页能看到每个角色的移速。
- 选择张总时，玩家能预期他较慢。
- 选择阿杰时，玩家能预期他较快。
- 相同角色左右移动速度一致。
- 不同角色速度差异来自配置，不来自方向或站位。

## 6. Bug 4：远程攻击不管往哪边走都只打一边

### 6.1 现象

远程技能方向不符合玩家当前输入方向，例如往左走但远程攻击仍向右，或不管怎么移动都打向同一侧。

### 6.2 历史根因

远程弹道方向取决于 `attacker.facingRight`。旧逻辑中 `facingRight` 不是由输入决定，而是由双方相对位置自动决定：

- 玩家在对手左边时，自动面向右。
- 玩家在对手右边时，自动面向左。

这导致玩家虽然按了某个方向，但系统下一帧可能又把朝向改回「面向对手」。远程攻击自然就会朝固定方向或朝对手方向发射。

### 6.3 当前代码状态

当前本地和服务端都已改为输入决定朝向：

- 刚按过左，远程技能向左。
- 刚按过右，远程技能向右。
- 原地不动时，保留最后一次移动朝向。

### 6.4 设计补充

如果后续希望保留「自动面向对手」的格斗体验，不能再直接复用 `facingRight` 作为技能方向。可以拆成两个概念：

```ts
facingRight: boolean;      // 视觉朝向，可自动面向对手
aimingRight: boolean;      // 攻击方向，由输入或锁定系统决定
lastInputRight: boolean;   // 最近一次水平输入方向
```

但以当前游戏形态，更简单稳定的方案是继续使用「输入决定朝向」。

### 6.5 验收标准

- 玩家1按 `A` 后释放远程技能，弹道向左。
- 玩家1按 `D` 后释放远程技能，弹道向右。
- 玩家2按 `ArrowLeft` 后释放远程技能，弹道向左。
- 玩家2按 `ArrowRight` 后释放远程技能，弹道向右。
- 双方交换站位后仍符合输入方向。
- 在线模式和本地模式表现一致。

## 7. 建议修复优先级

### P0：保持本地和服务端移动规则一致

当前已经完成核心修复：本地和服务端都取消了移动自动格挡减速，改为输入决定朝向；本地未使用的 `updateFacing(state)` 也已删除。

在线模式如果运行中仍看到旧表现，需要重启后端进程后重新验证。

### P1：补齐角色选择可回退能力

这是用户当前明确感受到的流程 bug。

已实现：

- 增加 `setSelectPhase` 或 `setSelectingSlot`。
- 玩家1、玩家2预览卡可点击切换编辑对象。
- 玩家2阶段展示「重新选择玩家1」。
- 返回选择页时明确保留还是重置状态。

### P1：在角色卡展示移速

这是「一个角色快一个角色慢」的主要解释成本。

已实现：

- 角色卡增加 `移速` 数值。
- 详情面板增加移速条。
- 详情面板根据移速显示「重装慢速 / 偏快 / 高机动 / 标准移速」提示。

### P2：重新设计格挡机制

如果还需要格挡，不建议继续绑定「远离对手移动」。

建议独立按键：

- 玩家1：`S` 或 `ShiftLeft` 格挡。
- 玩家2：`ArrowDown` 或 `ShiftRight` 格挡。

格挡逻辑应单独进入输入协议，在线模式需要同步到服务端。

## 10. 结论

当前问题不是单个视觉 bug，而是三个逻辑点叠加：

1. 角色选择流程是单向的，缺少回退编辑玩家1的入口。
2. 历史移动逻辑把「远离对手」绑定成格挡减速，造成左右移动时快时慢。
3. 不同角色本身 `moveSpeed` 差异明显，但 UI 没有展示移速，导致设计差异被误认为 bug。

当前移动和远程方向的核心逻辑已经按「输入决定朝向」方向修正。角色选择页的回退能力和移速展示也已落地，玩家现在能更清楚地理解并控制自己的选择。
